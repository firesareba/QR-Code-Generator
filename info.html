Data Settings

Base Patterns

These are the patterns that tell the scanner "I'm a QR Code". They are shown in Off-white and Grey. They come in 3 sub groups: Finder Patterns, Timing Strips, and Alignment Patterns
The Finder Patterns are the most recognizable, the 3 square shapes in the corners. These are used to define the orientation of the qr code relative to the scanner.
The Timing Strips are used to identify the rows and columns during scanning.
The Alignment Patterns appear in Versions 2 and above, and my understanding is they are used to correct for warping, if the paper curls or something. As such, there are more as the Version increases

Main Data

Before getting into the Data, we need to understand how the bitstream moves. Imagine splitting the grid into groups of 2 columns. Start in the bottom right corner. Once filled, move to the other cell in the same row, same column group. Then, move up to the right. If you hit an already filled square, skip it. Once you finish a column pair, go to the next column pair and reverse ONLY the y direction. So if you were going right to left, bottom to top, go right to left, top to bottom. NEVER go left to right. The Data is represented in Black and White This consists of 3 parts: Mode, Length, and Data
The Mode is a 4 bit identifier that tells the scanner how the data is encoded. The 4 modes are Binary(Used in my generator), Numeric, Alphanumeric, and Japanese Kanji.
The Length is 1 byte for Versions 1 through 9, and 2 bytes for Versions 10 and above. This stores how long the actual Data is. This will make more sense once you read Padding
The Data is the part that stores your url. It converts each character to its ASCII binary code, padding 0s till its 1 byte(8 bits). This comprises the bitstream. However, we do not write the bit stream in one long sequence. We break the code into blocks. Then, write byte 1 of block 1, byte 1 of block 2 and so on.

Padding

In every QR Code, there is a amount of space set aside for the data. This space is NOT dependent on the length of the data. We use padding to fill the leftover space.
First, we add the Terminators(Shown in Red). These just get the remaining data bits to a multiple of 8, so that bytes will fit evenly in it. There are at most 4.
The remaining space is filled with Padding Bytes(Shown in Purple). These are the numbers 17 and 236 represented in binary(and padded to 1 byte), alternating, in that order.

Error Correction

Error Correction (Shown in Green) accounts for if your QR Code is damaged. This way, it can still scan even if there is a stain or something.
Think of everything byte in the data space (including padding) as a number represented in binary. You can think of this as a polynomial. Now we can add some more terms to this polynomial so it is divisible by a generator polynomial. If a number is read wrong by the scanner, it will no longer be divisible. We can loop over every coefficient to see which one was incorrect.
To determine what the Error Correction bytes should be, we can use polynomial division. Divide the polynomial you get from data codewords by a generator polynomial. We want the remainder to be 0, so we just move the remainder to the other side. This means the remainder of Data Polynomial/Generator Polynomial is the Error Polynomail.
In bigger versions, the Data Polynomial is split into multiple blocks. These blocks are treated as individual polynomials which run the error correction algorithm individualy. That way, if a small part of the QR Code is damaged, we don't have to loop over ALL the coefficients, as that would be inefficient. Depending on the Correction Level, more of the qr code is reserved for error correction, and it can handle bigger stains. This is called Reed-Solomon Error Correction. Error Correction is NOT interleaved.

Maksing

Large patches of black or white can be interpreted as large error patches. To combat this, bits are reversed based on a pattern. All non-essential bits(data, padding, and error correction) are masked. Some patterns include a checkerboard, or every 3rd column.

Version Information

For Versions 7 and above, the timing strips are no longer enough to determine the Version number.
Version Information (Shown in Blue) stores the version as 6 padded bits in binary. Since this is critical information, it is stored identically in 2 places.
It also has error correction. However, it is not Reed-Solomon, the one that the main data uses. It uses Bose-Chaudhuri-Hocquenghem algorithm to expand the 6 bits into 18 BCH is based on Hamming distance.
Consider you only want to transmit 0 or 1. If a bit flip occurs, you will get the wrong answer. Instead, you represent it as 000 and 111, which are 3D coordinates. If a single bit flips, you were probably trying to transmit the corner which is closer to this new distance. Now take this concept and expand it to a 18 Dimensional object. Mathematically it is very simple.

Format Information

Format Strips (Shown in Orange) store the mask method and error correction level. Format Strips are very similar to Version info. They are stored in 2 spots, and use BCH error correction.
